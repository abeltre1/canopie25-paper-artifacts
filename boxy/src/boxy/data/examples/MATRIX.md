# boxy Example Matrix: every engine x runtime x scheduler

18 end-to-end combinations (2 engines x 3 runtimes x 3 schedulers). Every
`### Running Command` below is REAL boxy output (generated by dry-run via
`hack/gen_matrix.py`, not hand-written; volume paths show as $PWD/... for the
generating machine, and pulled-model paths as the dry-run placeholder
/path/to/model). The E2E steps are identical for every combo:

```bash
boxy examples export ./examples          # drop the packaged profiles into ./examples first
boxy pull  --box examples/boxes/<box>.toml                          # skip for path-based models
boxy build --box examples/boxes/<box>.toml --location examples/locations/<loc>.toml   # apptainer only (OCI->SIF)
boxy serve --box examples/boxes/<box>.toml --location examples/locations/<loc>.toml
curl -s http://<host>:<port>/v1/models                              # readiness
curl -s http://<host>:<port>/v1/chat/completions -H 'Content-Type: application/json' \
     -d '{"messages":[{"role":"user","content":"hello"}],"max_tokens":32}'
boxy bench --box examples/boxes/<box>.toml --batch-sizes 1,2,4,8 -o results.csv
boxy list && boxy stop --box examples/boxes/<box>.toml              # podman/docker lifecycle
```

Or skip the profiles entirely — the v2 front door resolves all of this from
the model alone (see README): `boxy serve <model> [--scheduler slurm --gpus N]`.

Engine notes: the llama.cpp box (`qwen-gguf`, port 8090) pulls a small GGUF
and runs on CPU anywhere (on ROCm/Intel GPU locations it auto-selects the
GPU-capable quay.io/ramalama images); the vLLM box (`vllm`, port 8000)
expects staged weights in `${MODELS_DIR}` and GPUs (cluster-class).
Scheduler notes: slurm/flux commands run from a login node; inside an
existing allocation use a location with `scheduler = "none"`. Apptainer
serve is foreground: stop with Ctrl-C or `scancel`/`flux cancel`.
